编写一个git提交帮助工具
核心功能
可以根据工作区暂存的代码变更，调用大模型生成提交日志

完整功能方案细化
1. 配置文件管理 (Configuration Management)
这是整个应用的核心，用于持久化用户数据。
存储位置: 配置文件将存储在用户的主目录下，例如 ~/.Commit-Pal/config.json。这样做的好处是，软件重装或升级后配置不会丢失。
文件格式: 采用 JSON 格式，因为它结构清晰，易于读写和扩展。
核心结构:
管理服务 (ConfigService):
应用启动时，自动加载 config.json 文件。
提供方法来读取、更新和保存配置。
API密钥需要进行简单的加密/解密处理，避免明文存储。
2. GUI 界面设计 (User Interface)
一个直观的GUI是提升用户体验的关键。我们可以设计一个三栏布局的界面。
主窗口布局 (三栏式):
| 项目列表 (左栏) | 代码变更 & 提交信息 (中栏) | 设置 & 提示词 (右栏) |
| ---------------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------- |
| - Project-A | 暂存区变更 (git diff --cached) | 全局模型设置 |
| - Project-B | - 一个只读、可滚动的文本区域，显示代码diff。 | - 模型提供商选择 (下拉菜单) |
| - [+] 添加项目 按钮 | - 刷新 按钮，重新获取变更。 | - API Key 输入框 (密码模式) |
| - [-] 删除项目 按钮 | 生成的提交信息 | - 模型名称 (如 gpt-4o) |
| | - 一个 可编辑 的多行文本输入框，用于显示和修改AI生成的信息。 | 项目特定提示词 (Prompt) |
| | - 生成提交信息 按钮。 | - 一个 可编辑 的多行文本框，用于编辑当前项目的提示词。 |
| | - 复制 按钮 | - 保存提示词 按钮 |
| | - 执行 Git Commit 按钮 (可选，但推荐) | |
3. 核心业务流程 (Core Workflows)
多项目管理流程:
添加项目:
用户点击 [+] 添加项目 按钮。
弹出文件浏览器，让用户选择一个本地的 文件夹。
程序检查该文件夹是否为有效的Git仓库（例如，是否存在 .git 目录）。
如果是，则为该项目生成一个唯一ID，使用文件夹名称作为项目名，并添加一个默认的提示词模板。
将新项目信息保存到 config.json 并刷新左侧项目列表。
切换项目:
用户在左侧列表中点击不同的项目。
程序更新 selected_project_id 配置。
中栏的 代码变更 区域自动刷新，显示新选定项目的 git diff --cached 内容。
右栏的 提示词 区域也更新为当前项目的 custom_prompt。
删除项目:
用户选择一个项目，点击 [-] 删除项目 按钮。
弹窗确认 "确定要从此列表中移除 [项目名] 吗？这不会删除你的本地文件。"
确认后，从 config.json 中移除该项目信息并刷新列表。
提交信息生成与编辑流程:
用户在左侧选择一个项目。
程序自动在中栏显示该项目的暂存区代码变更。
用户点击 生成提交信息 按钮。
程序读取右栏的 提示词 和中栏的 代码变更。
将两者组合成一个完整的Prompt，发送给 llm_settings 中配置的大模型API。
接收到AI返回的提交信息后，显示在中栏的 可编辑 文本框中。
用户可以直接在该文本框中对信息进行 任意修改。
修改满意后，可以点击 复制 按钮使用，或点击 执行 Git Commit 按钮直接完成提交。
自定义提示词流程:
用户选择一个项目。
在右栏的 项目特定提示词 文本框中，可以直接编辑内容。
编辑完成后，点击 保存提示词。
程序将更新后的提示词内容保存到 config.json 中对应项目的 custom_prompt 字段。
下次为此项目生成提交信息时，就会使用这个新的提示词。
